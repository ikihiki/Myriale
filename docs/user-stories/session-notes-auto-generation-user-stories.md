# session-notes-auto-generation-user-stories.md
## ノート（Lorebook）のAI半自動生成と通知

本ドキュメントは、セッション進行中にAIが人物・場所などのノート（Lorebook）を
半自動で生成・更新し、その差分をユーザーに通知して確認・採用できる体験を定義する。

前提用語
- Note: 人物/場所/組織/アイテム等の構造化ノート
- Canon: 確定情報
- Pending: 未確定の提案（ユーザー未承認）
- NoteUpdate: ノートの新規作成/更新案（差分）
- Notification: ユーザーに提示する更新通知
- Turn: 対話ログの単位

実装境界
- AI生成状態は `SessionExecution(kind=note-proposal)` で表す。
- 提案内容は `SessionArtifact(kind=note-patch)` と `SessionNoteProposal` に保存し、source Turnと根拠を必須にする。
- Apply/Edit then Applyはexpected Note revisionを検証し、`SessionNoteRevision`をappendする。
- Reject/SnoozeはCanonを変更しない。
- この基盤にはfixtureとreview UIを含むが、ノート用Prompt Builder、Provider adapter、AI Execution handlerは含まない。

---

## US-AN01: AIが重要情報を検出したらノートを自動作成してほしい

As a プレイヤー
I want 新しい人物や場所が登場したらAIがノートを自動作成してほしい
So that 手動入力の手間を減らしつつ重要情報を取り逃さない

トリガー
- 新規エンティティ（人物/場所/組織/アイテム等）の登場をAIが検出したとき
- 既存ノートにない固有名詞が一定の確度で重要と推定されたとき

期待される結果
- AIが「新規ノート案（Pending）」を生成する
- ノート案には、出典となる TurnId が紐づく
- ユーザーに通知が届く（US-AN03）

制約
- AIは自動的にCanon確定してはならない
- 既存の同名ノートがある場合は新規作成せず照合候補に回す

---

## US-AN02: AIが既存ノートを自動更新してほしい

As a プレイヤー
I want 既存人物・場所に新しい事実が判明したらAIがノート更新案を作ってほしい
So that 設定の一貫性を保ちながらノートを育てられる

トリガー
- 既存ノートに未記載の属性（外見・関係性・位置・状態など）が会話から判明したとき
- 既存の未確定（rumor/tentative）情報が裏付けられたとき
- 状態変化（負傷、感情、居場所、所持品など）が発生したとき

期待される結果
- AIが「更新差分（Pending NoteUpdate）」を生成する
- 更新差分には
  - 変更対象フィールド
  - 変更前/変更後
  - 変更根拠（TurnId）
  が含まれる
- ユーザーに通知が届く（US-AN03）

制約
- Canonの上書きは原則「要確認」とし、勝手に確定しない
- 矛盾がある場合は「矛盾検出通知」（US-AN06）に回す

---

## US-AN03: ノートの新規作成や更新があれば通知してほしい

As a プレイヤー
I want ノートの新規作成や更新が起きたら通知してほしい
So that 重要な変化を見逃さず、必要ならすぐ確認できる

通知の種類
- 新規ノート作成案（Pending）
- 既存ノート更新案（Pending）
- 矛盾検出（要判断）
- 重要度高の更新（関係性/正体/場所の確定など）

期待される結果
- セッション画面に「通知バッジ（件数）」が表示される
- 通知一覧で、各更新の概要が見られる
- 通知から該当ノート（差分ビュー）へ遷移できる

補足
- 通知はターンごとにまとめてもよい（スパム防止）

---

## US-AN04: 通知からノート更新案をレビューして採用/却下したい

As a プレイヤー
I want 更新案を差分で確認して採用/却下したい
So that AIの誤りを防ぎながらノートを整備できる

ユーザー行動
- 通知を開く
- 差分ビューを確認する
- 以下を選択する
  - 採用（Apply）
  - 一部採用（Edit then Apply）
  - 却下（Reject）
  - 保留（Snooze）

期待される結果
- 採用: ノートに反映され、必要ならCanon確定される
- 却下: 反映されず、理由（任意）を残せる
- 保留: 通知は未処理として残る

制約
- 自動確定は禁止
- 重要フィールド（名前/正体/関係性など）は確認を必須にできる（設定可能）

---

## US-AN05: ノート更新通知の頻度や粒度を設定したい

As a プレイヤー
I want 通知の頻度や対象を調整したい
So that プレイ中の邪魔にならない

設定例
- 通知タイミング
  - 毎ターン
  - 一定ターンごと
  - 章の終わり
  - 手動でまとめて確認
- 通知対象
  - 人物のみ
  - 場所のみ
  - 重要度高のみ
- 自動採用ポリシー（任意）
  - rumorのみ自動追加
  - Canonは必ず確認

期待される結果
- 設定に従って通知が抑制/集約される

---

## US-AN06: ノートと矛盾しそうな情報が出たら「判断が必要」と通知してほしい

As a プレイヤー
I want 矛盾が起きそうなときに通知してほしい
So that 勝手に設定が書き換わるのを防げる

トリガー例
- 人物の年齢・外見・所属の不一致
- 場所の位置関係の不一致
- 既に死亡した人物の再登場
- Canonフィールドへの上書き提案

期待される結果
- 矛盾の内容と根拠（TurnId）が提示される
- ユーザーが以下を選択できる
  - ノートを修正（Canon更新）
  - 今回の情報を誤りとして却下
  - 「噂/誤認」として未確定で保持

---

## US-AN07: AIがノートを使ってセッション要約（トークン削減用）を更新してほしい

As a プレイヤー
I want ノート更新と同時に要約も育てたい
So that 長編でも安定して進められる

期待される結果
- 章要約/状態要約が更新される（自動またはバッチ）
- 次ターン生成時のコンテキストが
  - Scenario Lore
  - Lorebook（Canon中心）
  - State（現在地/関係性/所持品/クエスト）
  - ChapterSummary
  - 直近Nターン
  で構成される

制約
- 要約も「自動確定」ではなく、必要に応じてユーザーが編集できる（任意）

## 実装メモ（設計指針）

ノート生成・更新の実行タイミング（推奨）
- ターン確定後に非同期で実行（Hangfire）
  - ExtractEntitiesFromTurn
  - ProposeNoteCreatesOrUpdates
  - CreateNotifications

通知UX（推奨）
- セッション画面右上に通知バッジ
- 通知一覧 → 差分ビュー → 採用/却下/保留

データ状態（推奨）
- Note は Canon と Pending を分離
  - Note: 確定データ
  - NoteUpdateProposal: Pending差分
  - Notification: 提案の提示単位

差分ビューに含める情報（最低限）
- 対象ノート名
- 変更フィールド
- Before / After
- 根拠ターン（TurnIdリンク）
- 重要度（low/medium/high）
