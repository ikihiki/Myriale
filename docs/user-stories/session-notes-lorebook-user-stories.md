# session-notes-lorebook-user-stories.md
## ノート（Lorebook）で人物・場所を詳細管理し、整合性担保とトークン削減に使う

本ドキュメントは、セッション中に蓄積されるノートを
「自由メモ」ではなく「構造化された百科事典（Lorebook）」として運用し、
AIのブレ防止（整合性担保）と、コンテキスト圧縮（トークン削減）に活用するための
ユーザーストーリーと設計要件を定義する。

前提用語
- Session: プレイ単位
- Turn: 対話ログの進行単位
- Lorebook Note: 構造化されたノート（人物・場所・組織・アイテム等）
- Canon: 正として扱う確定情報
- Summary: トークン削減のための要約（圧縮コンテキスト）

---

## US-L01: 人物ノートに詳細情報を登録したい

As a プレイヤー
I want 人物の詳細情報をノートに登録したい
So that 登場人物の設定が揺れず、後から参照できる

背景・意図
- AIが名前・年齢・口調・関係性を変えてしまうと没入感が壊れる
- 人物は「正（Canon）」として固定したい項目が多い

ユーザー行動
- 「人物」を新規作成する
- プロフィール項目を入力する（自由記述＋構造化）

期待される結果
- 人物ノートが作成され、セッションに紐づいて保存される
- ノートは検索・参照できる

構造（例）
- 表示名 / 別名
- 外見（特徴、服装、年齢感）
- 口調・話し方（敬語、方言、語尾）
- 性格（価値観、癖、弱点）
- 能力・役割（職業、スキル、立場）
- 関係性（主人公との関係、他人物との関係）
- 現在の状態（怪我、感情、居場所）
- 初出ターン（TurnId）
- 確定度（Canon / 未確定 / 噂）

---

## US-L02: 場所ノートに詳細情報を登録したい

As a プレイヤー
I want 場所の詳細情報をノートに登録したい
So that 地理や雰囲気がブレずに一貫する

ユーザー行動
- 「場所」を新規作成する
- 位置づけ・雰囲気・危険度などを入力する

期待される結果
- 場所ノートが保存され、以後の描写に参照される

構造（例）
- 名称 / 別名
- 種別（街、森、ダンジョン）
- 位置関係（近隣、距離、移動時間）
- 雰囲気（治安、文化、匂い、音）
- 主要NPC/施設
- 禁則・ルール（立ち入り禁止、法律）
- 初出ターン（TurnId）
- 確定度（Canon / 未確定 / 噂）

---

## US-L03: ノートに「確定情報（Canon）」と「未確定情報」を分けて管理したい

As a プレイヤー
I want 確定情報と未確定情報を分けたい
So that AIの参照優先度を制御できる

背景・意図
- 噂や推測が「事実」としてAIに固定されると破綻する

ユーザー行動
- 各フィールド、またはノート全体に確定度を付ける

期待される結果
- CanonはAIに強く参照される
- 未確定・噂は「可能性」として扱われる

---

## US-L04: AIにノート（Lorebook）を参照してほしい

As a プレイヤー
I want AIがLorebookを参照して語ってほしい
So that 設定のブレを防げる

期待される結果
- Narrative生成時に、関連する人物・場所ノートが参照される
- Canon情報が優先され、矛盾が減る

制約
- ノートは「世界のルール」ではなく、まずはセッション内の正史として扱う
- ただしCanonフラグの情報は矛盾させない

---

## US-L05: AIがノートと矛盾しそうなとき、勝手に変更せず確認してほしい

As a プレイヤー
I want 矛盾が起きそうなときに確認してほしい
So that 意図せず設定が書き換わらない

ユーザー行動
- 矛盾指摘が出たときに
  - ノートを更新する
  - AI出力を修正する
  - 噂（未確定）として扱う
  を選ぶ

期待される結果
- Canon情報が無断で変更されない
- 矛盾解消の意思決定がプレイヤー側にある

---

## US-L06: AIに人物・場所ノートの「追加候補」を提案してほしい

As a プレイヤー
I want AIにノート化すべき情報を提案してほしい
So that 重要情報の取りこぼしを防げる

期待される結果
- 新規登場人物や新規地点が出たら
  - ノート作成候補
  - 既存ノートへの追記候補
  が提示される

制約
- 自動確定しない（ユーザーが採用/編集/破棄）

---

## US-L07: Lorebookを「セッション要約（圧縮コンテキスト）」として使いたい

As a プレイヤー
I want Lorebookと要約を使ってトークンを減らしたい
So that コストを抑えつつ長編を安定運用できる

背景・意図
- 長文ログを毎回全投入すると高コストで破綻もしやすい
- 重要な事実だけを「固定辞書」と「要約」に圧縮したい

期待される結果
- セッション再開や次ターン生成時に
  - Lorebook（Canonのみ）
  - 重要状態（フラグ、所持品、関係性）
  - 章要約（Summary）
  - 直近Nターン
  の組み合わせでコンテキストを再構築できる

---

## US-L08: 章（Chapter）単位で要約を生成・更新したい

As a プレイヤー
I want 章単位の要約を持ちたい
So that 長編でも見通しが良く、トークンを節約できる

ユーザー行動
- 章の開始/終了を指定する（またはAIが提案する）
- 「章要約を生成」を実行する

期待される結果
- ChapterSummaryが保存される
- 次の生成では章要約が優先的に参照される

補足
- 「章」境界は
  - 場所移動
  - 目的達成
  - 大事件
  などで切ると良い

---

## US-L09: ノートを参照していることをUI上で可視化したい

As a プレイヤー
I want AIがどのノートを参照したか分かると嬉しい
So that 誤りの原因を特定しやすい

期待される結果
- そのターンで参照されたノート一覧が表示される（小さくでよい）
- クリックでノート編集に飛べる

---

## US-L10: ノートと要約の品質を維持するために「整合性チェック」をしたい

As a プレイヤー
I want ノートや要約の矛盾を検出したい
So that 参照情報が壊れないようにできる

期待される結果
- AIまたはルールで矛盾を検出し、候補を提示する
- 修正の確定はユーザーが行う

---

## 設計要件：コンテキスト圧縮（推奨構成）

次ターン生成時にAIへ渡すコンテキストは以下の優先順を推奨する。

1) Scenario Lore（シナリオ固定の世界観）
2) Lorebook（セッションのCanon辞書）
3) Session State（状態変数、関係性、所持品、現在地）
4) ChapterSummary（直近章の要約、複数あれば最新優先）
5) Recent Turns（直近Nターンの生ログ）

ポイント
- 生ログを減らし、LorebookとSummaryを厚くする
- 「未確定」情報はAIに断定させない（推測として扱わせる）
